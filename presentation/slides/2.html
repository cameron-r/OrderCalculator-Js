Anatomy of a story

User story template

  • As a [role]
  • I want [some feature]
  • So that [business value]

Acceptance criteria

  • Given [context]
  • When [action occurs]
  • Then [expected behaviour]

BDD is when we write scenarios with the behaviour of the system in mind and drive code (tests) from there

We write our tests similar to how a BA would write acceptance criteria

This means/results in some collaboration between developer and PM/PO/BA (or at least makes the dev put their BA hat on)
